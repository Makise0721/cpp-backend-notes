# 06 — 幂等性设计与数据一致性模型

---

## 1. 幂等性——重复操作，结果一致

**同一个操作执行一次和执行多次，效果相同。**

### 1.1 为什么需要幂等？

```
场景: 用户点了一次"下单"，但网络超时，前端重试了 3 次
→ 如果后端不幂等 → 用户被扣了 3 次钱！

原因:
- 网络重试（TCP 超时重传、HTTP 重试）
- MQ 消息重复投递（at-least-once）
- 用户多次点击
```

### 1.2 实现方式

#### 方式 1：唯一索引（数据库约束）

```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(64) UNIQUE,   -- 唯一约束
    -- ...
);

-- 第二次插入相同的 order_no → 唯一索引冲突 → 幂等
INSERT INTO orders (order_no, ...) VALUES ('ORD2024001', ...);
```

#### 方式 2：Token 机制

```
1. 客户端先请求 token: POST /getToken → server 返回 token
2. 客户端真正请求: POST /order + token
3. 服务端检查 token 是否已使用过
   ├─ 未使用 → 执行业务 → 标记 token 为已使用
   └─ 已使用 → 直接返回成功（幂等）
```

#### 方式 3：状态机

```
订单状态: PENDING → PAID → SHIPPING → COMPLETED

重复支付请求:
  第一次: PENDING → PAID ✅
  第二次: 状态已是 PAID → 不重复处理 ✅
```

#### 方式 4：乐观锁（版本号）

```sql
UPDATE account SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = 5;
-- 如果 version 已变成 6（被别的请求改了）→ 更新 0 行 → 幂等
```

### 1.3 各层幂等

| 层级 | 保证方式 |
|------|----------|
| 数据库 | 唯一索引、乐观锁、事务 |
| MQ Consumer | 业务幂等 + offset 提交 |
| HTTP API | Token / 请求 ID 去重 |
| RPC | 幂等框架（如 gRPC 的 idempotency key） |

## 2. 数据一致性模型

### 2.1 三种一致性

```
强一致 (Strong):
  写入后立刻读 → 一定能读到最新数据
  例: MySQL 单机、ZooKeeper
  代价: 性能差

最终一致 (Eventual):
  写入后 → 经过一段延迟 → 最终所有副本一致
  例: DNS、Cassandra、CDN
  特点: 高性能，但可能读到旧数据

因果一致 (Causal):
  有因果关系的操作顺序一致，无因果关系的可能乱序
  例: "发了帖子" → "朋友评论" — 评论一定在帖子之后看见
  比最终一致更强，比强一致更弱
```

### 2.2 一致性模型光谱

```
弱 ←──────────────────────────→ 强

最终一致   因果一致   顺序一致   线性一致   强一致
(可用优先)                              (一致优先)
```

### 2.3 实际场景

| 场景 | 需要什么一致性 | 为什么 |
|------|:---:|------|
| 银行转账 | **强一致** | 钱不能丢不能多 |
| 社交 feed | 最终一致 | 晚几秒看到没问题 |
| 库存扣减 | 强一致 | 超卖 = 灾难 |
| 浏览量计数 | 最终一致 | 少几个计数可接受 |
| 搜索引擎索引 | 最终一致 | 新页面几分钟后才搜到 |

## 3. 幂等 + 一致性在实际系统中的应用

```
订单支付流程:
1. 前端生成 order_token（防重复提交）
2. 后端收到请求 → 检查 token → 创建订单（唯一索引防重）
3. 调支付通道 → 支付回调可能重试
4. 回调处理：
   - 查订单状态（状态机防重）
   - 更新余额（乐观锁 + 版本号）
   - 发 MQ 消息（消费者业务幂等）
5. 每层都考虑了幂等 → 整个链路安全
```
