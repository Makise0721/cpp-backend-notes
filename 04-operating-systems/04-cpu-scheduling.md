## 1. 调度算法的评价标准

| 指标 | 含义 |
|------|------|
| **CPU 利用率** | CPU 忙的时间比例 |
| **吞吐量** | 单位时间完成的进程数 |
| **周转时间** | 进程从提交到完成的总时间 |
| **等待时间** | 进程在就绪队列中等待的时间 |
| **响应时间** | 从提交到第一次响应的间隔 |

## 2. 经典调度算法

### 2.1 FCFS（先来先服务）

非抢占。按到达顺序排队，一个跑完才跑下一个。

| 优点 | 缺点 |
|------|------|
| 简单、公平 | **护航效应**：长进程堵死后面的短进程 |
| 无饥饿 | 平均等待时间差 |

```
P1(24ms) → P2(3ms) → P3(3ms)
P2 的周转时间 = 24 + 3 = 27ms（白白等了 24ms！）
```

### 2.2 SJF（最短作业优先）

**非抢占**：选预计执行时间最短的进程先跑。

**抢占式（SRTF = Shortest Remaining Time First）**：如果新来的进程剩余时间更短，抢占当前进程。

| 优点 | 缺点 |
|------|------|
| 平均等待时间最优 | 需要预知执行时间（不现实） |
| 对短作业友好 | 长作业可能饥饿 |

### 2.3 优先级调度

每个进程有优先级，高的先跑。可以是抢占式或非抢占式。

**问题**：低优先级进程可能永远得不到 CPU → **饥饿**。

**解决**：**老化**——等待越久，优先级逐渐升高。

### 2.4 时间片轮转（Round Robin, RR）

每个进程轮流运行一个**时间片**（time quantum）。时间片用完 → 排到队尾。

```
时间片 = 4ms:
P1(24ms): ████│████│████│████│████│████
P2(3ms):  ░░░░│
P3(3ms):  ▓▓▓▓│

P2 和 P3 不会等 P1 跑完，3ms 就结束了
```

| 时间片太大 | 时间片太小 |
|-----------|-----------|
| 退化成 FCFS | 上下文切换开销占比大（频繁切换） |

经验值：**时间片 10-100ms**，保证 80% 的 CPU burst 在一个时间片内完成。

### 2.5 多级反馈队列（MLFQ）

**实际操作系统最常见的调度策略**。

```
多个就绪队列，优先级从上到下递减:

Q0 (最高优先级, 时间片 8ms)  ← 新进程先放这里
Q1 (中等优先级, 时间片 16ms) ← Q0 用完时间片降到这儿
Q2 (最低优先级, 时间片 ∞)    ← FIFO，只有上面空了才调度
```

**规则**：
1. 高优先级队列优先
2. 同优先级 RR
3. 新进程进最高优先级
4. 用完时间片 → 降级
5. 定期将所有进程提升回最高优先级（防止饥饿）

## 3. Linux CFS（完全公平调度器）

Linux 2.6.23 起默认调度器，替代了 O(1) 调度器。

### 3.1 核心思想：vruntime

**每个进程有一个 `vruntime`（虚拟运行时间），记录其"获得的 CPU 时间的加权值"。CFS 总是选 `vruntime` 最小的进程运行**——确保所有进程公平地获得 CPU。**

```c
// 简化概念
vruntime += 实际运行时间 × (1024 / weight)
// weight 由 nice 值决定：
// nice=0 → weight=1024（基准）
// nice=5 → weight≈335（低优先级，vruntime 增长更快 → 更容易被抢占）
// nice=-5 → weight≈3121（高优先级，vruntime 增长更慢 → 更不容易被抢占）
```

### 3.2 数据结构：红黑树

所有可运行进程按 `vruntime` 组织在一棵红黑树中。

```
红黑树（按 vruntime 排序）:
          [vr=50]
         /       \
    [vr=30]    [vr=80]
    /    \      /    \
[vr=10][vr=40][vr=60][vr=100]
   ↑
最左节点 = 下一个要运行的进程（vruntime 最小）

找最小：O(1)（缓存了最左节点）
插入/删除：O(log n)
```

### 3.3 CFS 关键参数

```bash
# 调度周期（所有可运行进程都获得一次 CPU 的时间）
$ cat /proc/sys/kernel/sched_latency_ns  # 默认 6ms（目标延迟）
$ cat /proc/sys/kernel/sched_min_granularity_ns  # 默认 0.75ms（最小时间片）

# 时间片 = sched_latency_ns / 可运行进程数
# 但不少于 min_granularity_ns
```

### 3.4 CFS vs O(1) vs MLFQ

| | MLFQ | O(1) | CFS |
|------|------|------|------|
| 核心数据结构 | 多级队列 | 两个优先位图数组 | 红黑树（vruntime） |
| 选择下一个 | 查最高非空队列 | O(1) 位图操作 | O(1)（最左节点缓存） |
| 公平性 | 需老化机制 | 近似的 | **精确公平** |
| Linux 版本 | — | 2.4 ~ 2.6.22 | 2.6.23+ |

## 4. 实时调度

Linux 也支持实时调度（`SCHED_FIFO` / `SCHED_RR`），优先级始终高于普通进程（`SCHED_OTHER` / `SCHED_NORMAL`）。

```bash
$ chrt -f 10 ./my_realtime_app  # 设置 SCHED_FIFO，优先级 10
```

## 小结

| 算法 | 抢占 | 优点 | 缺点 |
|------|:---:|------|------|
| FCFS | ❌ | 简单 | 护航效应 |
| SJF/SRTF | 可选 | 平均等待最优 | 需预知时间，长作业饥饿 |
| 优先级 | 可选 | 灵活 | 低优先级饥饿（需老化） |
| RR | ✅ | 公平、响应快 | 时间片选择关键 |
| MLFQ | ✅ | **实际最常用** | 参数调优复杂 |
| CFS | ✅ | **精确公平、红黑树 O(log n)** | — |
