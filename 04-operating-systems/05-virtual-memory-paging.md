# 05 — 虚拟内存、分页、多级页表、TLB、缺页异常

## 1. 为什么需要虚拟内存？

**三个核心目标**：
1. **隔离**：每个进程以为自己独占 4GB/128TB 地址空间
2. **安全**：进程 A 无法访问进程 B 的内存
3. **高效利用**：不常用的页换出到磁盘（swap），物理内存同时服务多个进程

---

## 2. 地址空间

```
32 位系统:
┌──────────────┐ 0xFFFFFFFF
│   内核空间    │ 1 GB（Linux 默认，可配 2G/3G）
├──────────────┤ 0xC0000000
│   用户空间    │ 3 GB
│  ┌────────┐  │
│  │  栈    ↓ │ 高地址向低地址增长
│  │  ...    │ │
│  │  堆    ↑ │ 低地址向高地址增长（brk/mmap）
│  │  .bss   │ │ 未初始化全局变量
│  │  .data  │ │ 已初始化全局变量
│  │  .text  │ │ 代码段
│  └────────┘  │
└──────────────┘ 0x00000000
```

64 位：用户空间可达 128 TB（Linux x86_64：47 位虚拟地址）。

## 3. 地址翻译全过程

```
虚拟地址（程序看到的）→ 物理地址（硬件实际用的）

虚拟地址结构（48 位，4 级页表）:
| 63-48 | 47-39 | 38-30 | 29-21 | 20-12 | 11-0 |
| 符号扩展 | PGD   | PUD   | PMD   | PTE    | offset |
   (全0/全1)  9 bits  9 bits  9 bits  9 bits  12 bits

翻译流程:
CR3 寄存器 → PGD 表 → PGD[PGD_idx] → PUD 表物理地址
                      → PUD[PUD_idx] → PMD 表物理地址
                      → PMD[PMD_idx] → PTE 表物理地址
                      → PTE[PTE_idx] → 物理页帧号
                                      + offset → 最终物理地址
```

```
虚拟地址: 0x7F...1234
          │
          ▼
┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
│ PGD  │──→│ PUD  │──→│ PMD  │──→│ PTE  │
│[idx] │   │[idx] │   │[idx] │   │[idx] │──→ 物理页帧号
└──────┘   └──────┘   └──────┘   └──────┘
                                          ↓
                                    物理地址 = 页帧号 << 12 | offset
```

## 4. 页表项（PTE）结构

```
PTE（8 字节，64 位）:
| 物理页帧号 (40 bits) | flags (12 bits) | ... |

关键标志位:
┌──────────────┬──────────────────────────────┐
│ Present (P)  │ 该页在物理内存中？（1=是）     │
│ Read/Write   │ 是否可写                      │
│ User/Super   │ 用户态是否可访问               │
│ Accessed (A) │ 是否被访问过（用于 LRU 近似）   │
│ Dirty (D)    │ 是否被写入过（换出时需写回磁盘） │
│ NX           │ 禁止执行（No eXecute）         │
└──────────────┴──────────────────────────────┘
```

## 5. TLB——地址翻译的快速缓存

**没有 TLB，每次内存访问都需要 4 次页表查询（4 次内存访问！）。TLB 在 CPU 内部缓存了"虚拟页号 → 物理页帧号"的映射。**

```
CPU 取指/访存:
  → 查 TLB（虚拟页号 → 物理帧号）
     ├─ 命中 → 直接获得物理地址（~1 cycle）
     └─ 未命中（TLB miss）
        → 走页表查询（4 次内存访问）
        → 结果填入 TLB
        → 重试原操作
```

### 5.1 TLB 刷新

**进程切换时**：地址空间变了，TLB 中的旧映射全无效 → **必须刷新**。

```x86asm
mov rax, cr3
mov cr3, rax    ; 重写 CR3 → 刷新所有 TLB 条目（除 global 页外）

invlpg [addr]   ; 只刷新指定地址的 TLB 条目（更精细）
```

**这就是线程切换比进程切换快的主要原因之一——线程共享地址空间，不刷 TLB。**

### 5.2 TLB Shootdown

多核系统中，一个核修改了页表（比如 `munmap`），需要通知其他核刷新它们各自的 TLB → **TLB shootdown**，通过 **IPI（核间中断）** 实现。

## 6. 缺页异常（Page Fault）处理流程

```
进程访问虚拟地址 → MMU 查 TLB → miss → 查页表 → PTE.Present = 0
→ 触发缺页异常（#PF，14 号中断）

内核处理:
1. 检查地址是否合法（是否在进程的地址空间内）
   ├─ 非法 → SIGSEGV（段错误）
   └─ 合法 →
2. 找到一个空闲物理页框（或触发页面置换）
3. 从磁盘（可执行文件 / swap）读取数据到该物理页
4. 更新 PTE：设置物理地址 + Present=1
5. 返回用户态，重试引发缺页的指令
```

### 6.1 三类缺页

| 类型 | 场景 | 处理 |
|------|------|------|
| **Major fault** | 数据在磁盘（还未读入） | 磁盘 I/O（慢！） |
| **Minor fault** | 数据已在内存（如 COW、共享库已加载） | 仅建页表映射（快） |
| **Invalid fault** | 访问非法地址 | SIGSEGV |

```bash
# 查看进程缺页统计
$ ps -o min_flt,maj_flt,cmd -p <PID>
```

## 7. 大页内存（Huge Pages）

默认页大小 4KB。对于大内存应用，TLB 条目不够用 → 频繁 TLB miss。

**解决方案**：用 2MB（或 1GB）的页代替 4KB 的页，一个 TLB 条目覆盖更大范围。

```
4KB 页：覆盖 1GB 需要 262144 个 TLB 条目
2MB 页：覆盖 1GB 只需 512 个 TLB 条目 → TLB miss 大幅降低
```

```bash
# 查看大页使用情况
$ cat /proc/meminfo | grep Huge
```

数据库（MySQL/PostgreSQL）、AI 推理（vLLM）、DPDK 等场景大量使用大页。

---

## 小结

| 概念 | 一句话 |
|------|--------|
| 虚拟内存 | 每个进程独享地址空间，物理内存由内核按需映射 |
| 4 级页表 | PGD→PUD→PMD→PTE，每级 9 bit 索引 |
| TLB | 页表缓存，命中 ~1 cycle，miss 走 4 次内存访问 |
| 缺页 | 访问未映射的地址 → 内核分配物理页 + 读磁盘 |
| 大页 2MB/1GB | 减少 TLB miss，数据库和 AI 推理常用 |
