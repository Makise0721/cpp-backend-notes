# 02 — 线程：TCB、用户态/内核态线程、NPTL、进程 vs 线程

> 关联篇目：[01 进程基础](./01-process-basics.md) | [04 CPU 调度](./04-cpu-scheduling.md)

---

## 1. 什么是线程？

**线程是 CPU 调度的基本单位**。一个进程可以有多个线程，它们共享进程的大部分资源，但各自拥有独立的执行流。

```
单进程单线程:              单进程多线程:
┌─────────────┐           ┌─────────────────────┐
│   Process    │           │       Process        │
│  ┌────────┐  │           │  T1    T2    T3     │
│  │ Thread  │  │           │  │     │     │      │
│  │ (栈+PC) │  │           │  ▼     ▼     ▼      │
│  └────────┘  │           │ 共享: 代码/数据/堆/fd  │
│  独立资源     │           │ 私有: 栈/寄存器/TLS   │
└─────────────┘           └─────────────────────┘
```

---

## 2. 线程共享 vs 私有资源

| 共享（进程内所有线程） | 私有（每个线程独立） |
|-----------------------|---------------------|
| 代码段（`.text`） | 栈（stack） |
| 数据段（`.data`, `.bss`） | 寄存器（包括 PC/SP） |
| 堆（heap） | 线程局部存储（TLS） |
| 文件描述符表 | 信号掩码 |
| 信号处理函数 | errno |
| 工作目录、uid/gid | 线程 ID（TID） |

**同一进程的线程之间没有内存保护**——一个线程写野指针可以破坏另一个线程的栈。这是线程编程最大的安全隐患。

## 3. 用户态线程 vs 内核态线程

### 3.1 用户态线程（绿色线程）

```
用户态线程（如早期的 Java Green Threads、Go 的 goroutine 的前身）:
┌──────────────────────────────┐
│         用户态线程库           │
│  T1    T2    T3  ← 用户调度   │
│  └─────┼─────┘               │
│        │ 映射到一个内核线程    │
│  ┌─────┴─────┐               │
│  │ 内核线程   │ ← 内核看到的   │
│  └───────────┘               │
└──────────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| 切换极快（不陷内核） | 一个阻塞全阻塞（内核只看到一个线程） |
| 不需要内核支持 | 无法利用多核 |

### 3.2 内核态线程（1:1 模型）

```
每个用户线程对应一个内核线程（Linux NPTL 用此模型）:
T1 → K1
T2 → K2
T3 → K3
```

| 优点 | 缺点 |
|------|------|
| 一个阻塞不影响其他 | 创建/切换代价较高（需要系统调用） |
| 可以利用多核 | 数量受内核限制 |

### 3.3 混合模型（M:N）

Go 的 GMP 调度模型是 M:N 的现代范例——M 个 goroutine 运行在 N 个内核线程上。

## 4. Linux 线程实现：NPTL

**NPTL** = Native POSIX Thread Library。Linux 2.6 起使用的 1:1 线程模型。

### 4.1 核心原理

```
Linux 用轻量级进程（LWP）实现线程:
- fork() 创建进程（新 PID + 独立地址空间）
- clone() 创建线程（共享地址空间，但有自己的 TID）

clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | ...)
      ↑共享地址空间 ↑共享工作目录 ↑共享fd表    ↑共享信号处理
```

```bash
# Linux 中进程和线程都是 task，只是资源共享程度不同
$ ps -eLf  # -L 显示线程
UID  PID  PPID   LWP  C  NLWP  CMD
root 1234    1  1234  0    4   nginx: master
root 1234    1  1235  0    4   nginx: worker  ← 4 个线程（NLWP=4）
root 1234    1  1236  0    4   nginx: worker
root 1234    1  1237  0    4   nginx: worker
```

- `PID` = 进程 ID（实际是 tgid，线程组 ID）
- `LWP` = 轻量级进程 ID（每个线程的唯一 ID，`gettid()` 返回值）
- `NLWP` = 线程数

### 4.2 NPTL 特性

- 基于 1:1 模型：每个用户线程对应一个内核线程
- 支持线程本地存储（TLS → `__thread` 或 C++11 `thread_local`）
- 支持 `futex`（快速用户空间互斥锁）——性能关键

## 5. 进程 vs 线程——全面对比

| 维度 | 进程 | 线程 |
|------|------|------|
| **资源** | 独立资源（地址空间、fd 表） | 共享进程资源 |
| **调度** | 资源分配基本单位 | **CPU 调度基本单位** |
| **创建开销** | 大（复制页表、分配新地址空间） | 小（共享地址空间） |
| **切换开销** | 大（切页表、刷 TLB） | 小（只切寄存器+栈） |
| **通信** | 需要 IPC（管道、共享内存等） | 直接读写共享变量 |
| **安全性** | 一个崩溃不影响其他 | 一个崩溃全进程崩溃 |
| **多核利用** | ✅ | ✅ |

```
场景选择：
├─ 需要强隔离 → 多进程（Chrome 每个 tab 一个进程）
├─ 需要高并发 + 共享数据 → 多线程（Web 服务器 worker 线程）
└─ 两者兼需 → 进程 + 线程（nginx: 多进程 worker + 每个 worker 可多线程）
```

## 小结

| 概念 | 一句话 |
|------|--------|
| 线程 | CPU 调度基本单位，共享进程资源，私有栈和寄存器 |
| 用户态线程 | 内核无感知、切换快、一个阻塞全阻塞 |
| 内核态线程 | 1:1 映射、可多核、创建有开销 |
| NPTL | Linux 1:1 线程实现，基于 `clone()` + `futex` |
| 进程 vs 线程 | 隔离性 ↔ 共享性 的权衡 |
