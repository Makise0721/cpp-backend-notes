# 01 — CAP 定理与一致性哈希

---

## 1. CAP 定理——分布式系统的三角困境

**Consistency（一致性）、Availability（可用性）、Partition Tolerance（分区容错）三者只能同时满足两个。**

```
       C (一致性)
       /\
      /  \
     /    \
    /  CA  \      ← 传统单机数据库 (MySQL)
   /        \
  A ──────── P    ← CA 不可能同时满足 P
 (可用性)  (分区容错)

实际选择：既然网络分区 P 无法避免，只能在 CP 和 AP 之间权衡
```

### 1.1 三种组合

| 组合 | 含义 | 例子 | 牺牲 |
|------|------|------|:---:|
| **CP** | 分区时保证一致性，牺牲可用性 | ZooKeeper、Etcd | A |
| **AP** | 分区时保证可用性，牺牲一致性 | Cassandra、DynamoDB | C |
| **CA** | 无分区时可用且一致 | MySQL（单机） | P |

### 1.2 实际理解

CAP 不是"二选一放弃第三个"，而是**当网络分区发生时，你选择一致还是可用**：

```
CP (ZooKeeper):
  网络分区 → 少数派节点拒绝服务 → 用户端报错
  → 牺牲了 A，保证了 C

AP (Cassandra):
  网络分区 → 所有节点继续接受写入 → 数据可能不一致
  → 牺牲了 C，保证了 A（后续通过反熵修复）
```

## 2. PACELC——CAP 的扩展

```
出现网络分区 (P):
  ├─ 选 A (可用)
  └─ 选 C (一致)

无网络分区 (E: Else):
  ├─ 选 L (Latency — 低延迟)
  └─ 选 C (Consistency — 强一致)
```

## 3. 一致性哈希——解决分布式缓存/存储的扩容问题

### 3.1 问题：普通哈希的扩容灾难

```
普通哈希: node = hash(key) % N

N=3:  key1→node0, key2→node1, key3→node2
N=4:  key1→node1, key2→node2, key3→node0  ← 几乎所有 key 都换了节点！
→ 缓存全部失效 → 大量请求穿透到 DB → 雪崩
```

### 3.2 一致性哈希原理

```
1. 将哈希空间视为一个环：0 ~ 2³²-1

   0/2³²
    ↓
  ┌──────────┐
  │          │     节点 A 在位置 100
  │    B     │     节点 B 在位置 5000
  │          │     节点 C 在位置 20000
  │  A   C   │
  └──────────┘

2. key 映射到环上顺时针方向的第一个节点

3. 增删节点只影响相邻节点之间的数据:
   删除节点 B → 只有 B 负责的 key 需要迁移到 C
   增加节点 D → 只有 D 顺时针下一个节点 C 的部分数据需要迁移给 D
```

### 3.3 虚拟节点——解决数据倾斜

```
没有虚拟节点:
  3 个物理节点在环上的位置可能不均匀 → 数据倾斜

加入虚拟节点:
  Node A → A1, A2, A3, ..., A100  (100 个虚拟节点)
  Node B → B1, B2, B3, ..., B100
  虚拟节点均匀分布在环上 → 数据均匀

增加物理节点:
  Node D → D1, D2, ..., D100
  从现有虚拟节点的后继中拿数据 → 均衡迁移
```

### 3.4 实际应用

- **Redis Cluster**：16384 个 slot，每个节点负责一部分 slot
- **Nginx 负载均衡**：`hash $request_uri consistent`
- **Dubbo 服务路由**
- **CDN 缓存调度**

## 小结

| 概念 | 一句话 |
|------|--------|
| CAP | 分区时选 C(一致) 还是 A(可用)，P 无法避免 |
| CP 系统 | ZK/Etcd — 分区时拒绝服务 |
| AP 系统 | Cassandra — 分区时继续服务，后续修复 |
| 一致性哈希 | 环 + 虚拟节点，扩容只迁移最少数据 |
| 虚拟节点 | 解决物理节点分布不均的问题 |
