## 1. 死锁——四个必要条件

```
死锁 = 两个或多个线程互相等待对方释放资源，永远无法前进
```

### 1.1 四大必要条件（缺一不可）

| 条件 | 含义 | 如何打破 |
|------|------|----------|
| **互斥** | 资源只能被一个线程占用 | 用无锁数据结构 / 原子操作替代 |
| **持有并等待** | 持有资源的同时等待其他资源 | 一次性申请所有资源（`scoped_lock`） |
| **不可剥夺** | 已分配的资源不能被强制拿走 | 用 `try_lock` + 超时放弃 |
| **循环等待** | 线程间形成资源等待环 | 统一加锁顺序 |

### 1.2 预防死锁的四种策略

```cpp
// 策略 1：统一加锁顺序（最实用）
// ❌ 线程 1: lock(A); lock(B);
// ❌ 线程 2: lock(B); lock(A);  ← 可能死锁
// ✅ 所有线程都按 A→B 的顺序加锁

// 策略 2：一次性加锁（用 scoped_lock / std::lock）
std::scoped_lock lock(m1, m2, m3);

// 策略 3：try_lock + 回退
while (true) {
    m1.lock();
    if (m2.try_lock()) break;
    m1.unlock();          // 回退，释放已持有的锁
    std::this_thread::yield();
}

// 策略 4：减少锁的持有时间 + 避免嵌套锁
```

### 1.3 死锁检测

**资源分配图**：节点 = 线程 / 资源，边 = 请求 / 持有。如果图中出现环 → 死锁。

### 1.4 银行家算法（死锁避免）

**在分配资源前，先判断分配后系统是否仍处于安全状态**。只有当存在一个安全序列（所有进程都能完成）时才分配。

## 2. 活锁（Livelock）

两个线程都在"让步"以避免冲突，但让步的动作本身导致谁也推进不了。

```
线程 A: 看到 B 在用资源 → 释放自己的锁 → 等一下
线程 B: 看到 A 在等 → 也释放自己的锁 → 等一下
线程 A: 重新尝试 → 又看到 B...
→ 两人都在动，但谁也没进展
```

**解决方案**：引入随机退避（random backoff）或优先级。

---

## 3. 线程饥饿（Starvation）

高优先级线程一直占据资源，低优先级线程永远得不到执行。

**解决方案**：老化（aging）——等待时间越长优先级越高。

## 4. 优先级反转与优先级继承

### 4.1 问题

```
高优先级 Task H → 等待锁 L（被低优先级 Task L 持有）
低优先级 Task L → 持有锁 L，但被中优先级 Task M 抢占
→ Task H 被 Task M 无限期阻塞（即使 M 优先级比 H 低）
→ 优先级反转！
```

**火星探路者号就因为这个 bug 导致系统不断重启。**

### 4.2 优先级继承

持有锁的低优先级线程**临时继承**等待该锁的最高优先级：

```
Task L 持有锁 → Task H 来等待 → Task L 继承 H 的优先级
→ Task L 不会被 Task M 抢占了 → 快速释放锁 → Task H 恢复运行
```

### 4.3 优先级天花板协议

给每把锁设置一个"天花板优先级"（= 所有可能使用该锁的线程中的最高优先级）。持有锁的线程立即提升到天花板优先级。

## 5. 惊群效应（Thundering Herd）

### 5.1 问题

多个线程同时阻塞在同一个事件上，事件发生后全部被唤醒，但只有一个能处理 → 其他线程白醒了 → 浪费 CPU。

```
经典：多个进程/线程 epoll_wait/accept 同一个 fd
连接到来 → 所有等待者全被唤醒 → 只有一个 accept 成功 → 其他继续睡
→ 大量无谓的上下文切换
```

### 5.2 解决方案

- **`accept` 惊群**：Linux 4.5+ 已在内核层面解决（`SO_REUSEPORT` + 轮转分配）
- **`epoll` 惊群**：使用 `EPOLLEXCLUSIVE` 标志（Linux 4.5+），只唤醒一个等待者
- **条件变量**：用 `notify_one` 而不是 `notify_all`（如果每任务只需一个处理器）

## 6. 锁优化策略

```
锁优化的五个层次：
1. 减小临界区 → 锁外做准备工作，锁内只做必须互斥的操作
2. 读写分离 → 读多用 shared_mutex，写少用 mutex
3. 分段锁 → ConcurrentHashMap（每个 bucket 一把锁）
4. 减少锁持有时间 → 用条件变量让渡 CPU，而不是持锁等待
5. 无锁替代 → atomic + CAS 替代 mutex
```

### 6.1 分段锁示例

```cpp
// 一个全局 mutex → 所有 bucket 竞争同一把锁 → 瓶颈
// 分段锁：每个 bucket 独立一把锁
class ConcurrentHashMap {
    struct Bucket {
        std::shared_mutex mtx;
        std::list<std::pair<int, int>> data;
    };
    std::vector<Bucket> buckets_;

    Bucket& getBucket(int key) {
        return buckets_[std::hash<int>{}(key) % buckets_.size()];
    }
    // 不同 bucket 的操作完全并行
};
```

## 小结

| 问题 | 原因 | 解法 |
|------|------|------|
| 死锁 | 四大条件同时满足 | 统一加锁顺序 / `scoped_lock` |
| 活锁 | 互相让步 | 随机退避 |
| 饥饿 | 高优先级抢占 | 老化 |
| 优先级反转 | 低优先级持锁阻塞高优先级 | 优先级继承 |
| 惊群 | 所有等待者被同时唤醒 | `EPOLLEXCLUSIVE` / `notify_one` |
