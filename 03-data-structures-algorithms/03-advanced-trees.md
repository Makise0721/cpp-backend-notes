## 1. B 树——为磁盘而生的多路搜索树

### 1.1 为什么需要 B 树？

二叉树的节点只存 1 个 key，树高 O(log₂n)。当数据在磁盘上时，每次访问一个节点 = 一次磁盘 I/O，代价极高。B 树的思路：**让每个节点存多个 key，降低树高，减少磁盘 I/O**。

```
B 树节点（m 阶，即最多 m 个孩子）:
┌─────┬─────┬─────┬─────┬─────┐
│ P0  │ K1  │ P1  │ K2  │ P2  │ ...（一个节点存多个 key + 多个孩子指针）
└─────┴─────┴─────┴─────┴─────┘
```

### 1.2 性质（m 阶 B 树）

- 每个节点最多 m 个孩子、m-1 个 key
- 除根外，每个节点至少 ⌈m/2⌉ 个孩子
- 根至少 2 个孩子（非叶子时）
- **所有叶子在同一层**（完美平衡）

```
5 阶 B 树示例（节点最多 4 个 key，5 个孩子）:
              [20, 40]
          /      |       \
    [5,10]     [25,30,35]    [50,60,70,80]
```

### 1.3 操作复杂度

| 操作 | 复杂度 |
|------|:---:|
| 查找 | O(log n) |
| 插入 | O(log n)（可能分裂节点） |
| 删除 | O(log n)（可能合并节点） |

## 2. B+ 树——数据库索引的核心

**B+ 树是 B 树的变种，所有数据存在叶子节点，非叶子只存索引。**

```
B+ 树:
                [20, 40]              ← 非叶子：只存索引（路由作用）
             /      |       \
        [5,10]  [25,30,35]  [50,60]   ← 叶子：存全部数据
           ↓       ↓        ↓
        数据块    数据块    数据块
           ↔       ↔        ↔         ← 叶子节点用链表串联（支持范围查询！）
```

### 2.1 B 树 vs B+ 树

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点都存 | 只在叶子存 |
| 叶子链表 | 无 | ✅（范围查询 O(log n + k)） |
| 非叶子节点 | 存 key + data | 只存 key（一个节点能存更多 key → 树更矮） |
| 查找性能 | 可能中途命中 | 必须走到叶子（更稳定） |
| 典型用途 | 文件系统 | **数据库索引（MySQL InnoDB）** |

**为什么 MySQL 用 B+ 树？**

1. **范围查询**：叶子链表支持 `WHERE id BETWEEN 100 AND 200` 的高效扫描
2. **树更矮**：非叶子只存 key 不存 data，一个 16KB 页能存更多索引项
3. **查询稳定**：每次都必须走到叶子，I/O 次数可预测

## 3. 哈夫曼树——最优前缀编码

用于**数据压缩**。核心思想：**高频字符用短编码，低频字符用长编码**。

### 3.1 构建过程

```
字符频率: A:5  B:9  C:12  D:13  E:16  F:45

1. 取两个最小频率：A(5) + B(9) → 新节点 14
2. 取两个最小频率：C(12) + D(13) → 新节点 25
3. 取两个最小频率：新14 + E(16) → 新节点 30
4. 取两个最小频率：新25 + 新30 → 新节点 55
5. 取两个最小频率：F(45) + 新55 → 根节点 100

结果:
          100
         /    \
       F:45    55
              /  \
            25    30
           / \   / \
         C:12 D 14  E:16
               / \
             A:5 B:9
```

### 3.2 编码规则

从根到叶子：**左分支 = 0，右分支 = 1**。

| 字符 | 编码 |
|------|------|
| F | 0 |
| C | 100 |
| D | 101 |
| A | 1100 |
| B | 1101 |
| E | 111 |

**哈夫曼编码是前缀码**：没有任何一个编码是另一个的前缀，因此不需要分隔符。

## 4. 字典树（Trie）——前缀匹配的利器

### 4.1 结构

```
单词: "cat", "car", "dog", "do"

        root
       /    \
      c      d
      |      |
      a      o
     / \    / \
    t*  r* g*  (empty)
```

- 每个节点代表一个字符
- 从根到某节点路径 = 一个前缀
- `*` 标记表示该节点是一个完整单词的结尾

### 4.2 基本实现

```cpp
class Trie {
    struct Node {
        Node* children[26] = {};
        bool isEnd = false;
    };
    Node root_;
public:
    void insert(const string& word) {
        Node* node = &root_;
        for (char c : word) {
            int idx = c - 'a';
            if (!node->children[idx])
                node->children[idx] = new Node();
            node = node->children[idx];
        }
        node->isEnd = true;
    }

    bool search(const string& word) {
        Node* node = &root_;
        for (char c : word) {
            int idx = c - 'a';
            if (!node->children[idx]) return false;
            node = node->children[idx];
        }
        return node->isEnd;
    }

    bool startsWith(const string& prefix) {
        Node* node = &root_;
        for (char c : prefix) {
            int idx = c - 'a';
            if (!node->children[idx]) return false;
            node = node->children[idx];
        }
        return true;
    }
};
```

| 操作 | 复杂度 |
|------|:---:|
| 插入 | O(L) → L = 单词长度 |
| 查找 | O(L) |
| 前缀搜索 | O(L) |

### 4.3 应用场景

- **自动补全**：输入前缀 → 找到该节点 → DFS 收集所有完整单词
- **拼写检查**：快速判断单词是否存在
- **IP 路由最长前缀匹配**
- **AC 自动机**：多模式串匹配（见 [10 字符串算法](./10-string-algorithms.md)）

## 5. 跳表（Skip List）——概率型平衡结构

### 5.1 为什么需要跳表？

平衡树（AVL / 红黑树）实现复杂，旋转逻辑容易出错。跳表用**概率**替代**旋转**，实现同样 O(log n) 的查找性能，但代码简单得多。

### 5.2 结构

```
Level 3:  head ─────────────────────→ 9 ──→ null
Level 2:  head ──────→ 3 ──────────→ 9 ──→ null
Level 1:  head ──→ 1 ──→ 3 ──→ 6 ──→ 9 ──→ null
Level 0:  head → 1 → 2 → 3 → 5 → 6 → 7 → 9 → null
                  ↑
             最底层是完整链表，每往上一层跳过更多元素
```

每层都是一个有序链表，高层链表是低层的"快速通道"。每个节点以概率 p（通常 0.5）决定是否出现在上一层。

### 5.3 查找过程

```
查找 6：
Level 3: head→9(>6，不前进，下到 Level 2)
Level 2: head→3(≤6)→9(>6，下到 Level 1)
Level 1: 从 3→6(找到！)
```

| 操作 | 平均 | 最坏 |
|------|:---:|:---:|
| 查找 | O(log n) | O(n) |
| 插入 | O(log n) | O(n) |
| 删除 | O(log n) | O(n) |
| 空间 | O(n) 期望 | O(n log n) |

### 5.4 跳表 vs 红黑树

| 特性 | 跳表 | 红黑树 |
|------|------|--------|
| 实现复杂度 | 低 | 高（旋转 case 多） |
| 范围查询 | O(log n + k)，天然支持 | 中序遍历 |
| 并发友好 | ✅（局部修改，锁粒度小） | 难（旋转影响大范围） |
| 内存 | 指针多（平均 2n 个指针） | 每个节点 3 个指针 + 颜色位 |
| 实际应用 | **Redis ZSet**、LevelDB | C++ std::map |

**Redis 选择跳表实现 ZSet**：实现简单 + 支持范围查询（ZRANGE）+ 并发友好。

## 小结

| 结构 | 核心思想 | 查找 | 典型用途 |
|------|----------|:---:|------|
| B 树 | 多路搜索，降低树高 | O(log n) | 文件系统 |
| B+ 树 | 数据只存叶子 + 链表 | O(log n) | 数据库索引 |
| 哈夫曼树 | 高频短码、低频长码 | — | 数据压缩 |
| Trie | 按字符前缀组织 | O(L) | 自动补全、拼写检查 |
| 跳表 | 多层索引 + 概率平衡 | O(log n) | Redis ZSet |
