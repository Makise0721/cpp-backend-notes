## 1. 缓存三问题

### 1.1 缓存穿透

**查询一个数据库中根本不存在的 key** → 每次穿透缓存打到 DB。

```
防御手段:
1. 布隆过滤器: 提前过滤掉一定不存在的 key
2. 缓存空值: 将 "key → null" 也缓存（短过期时间）
3. 参数校验: 在业务层拦截非法 key
```

### 1.2 缓存击穿

**热点 key 过期瞬间，大量请求同时打到 DB**。

```
防御手段:
1. 互斥锁: 第一个请求去加载 DB，其他等待锁释放后读缓存
2. 逻辑过期: value 中存过期时间，发现过期后异步更新，旧数据继续服务
3. 永不过期: 热点 key 不设过期时间，用异步任务更新
```

### 1.3 缓存雪崩

**大量 key 同时过期，或 Redis 集群宕机** → DB 瞬间承受所有流量。

```
防御手段:
1. 过期时间加随机值: EXPIRE key (3600 + random(0, 600))
2. 多级缓存: 本地缓存 (Caffeine) + Redis + DB
3. Redis 高可用: 哨兵/集群 + 熔断降级
4. 限流: 限制打到 DB 的流量
```

## 2. 缓存一致性策略

| 策略 | 方式 | 一致性 | 性能 |
|------|------|:---:|:---:|
| **Cache Aside** | 先更新 DB → 再删除缓存 | 最终一致 | 高 |
| **Read/Write Through** | 缓存层负责同步 DB | 强/最终 | 中 |
| **Write Behind** | 先写缓存 → 异步刷 DB | 弱（可能丢数据） | 极高 |

### 2.1 Cache Aside（最常用）

```
读:
  查缓存 → 命中返回
        → 未命中 → 查 DB → 写入缓存 → 返回

写:
  更新 DB → 删除缓存（为什么是删除而不是更新？）

为什么删除而不是更新？
  - 如果每次更新 DB 都同步更新缓存 → 可能更新了一个永远不会被读的缓存
  - 删除后，下次读自然会查 DB 并回填最新值
  - 删除操作比更新简单，无并发问题
```

### 2.2 延迟双删

```
写操作:
1. 删除缓存
2. 更新 DB
3. 延迟一小段时间（如 500ms）
4. 再次删除缓存（防止步骤 2 期间有读请求把旧值写入了缓存）
```

## 3. 淘汰策略

当 Redis 内存用完时，按什么规则淘汰 key：

| 策略 | 行为 |
|------|------|
| **noeviction** | 不淘汰，写操作直接报错（默认） |
| **allkeys-lru** | 所有 key 中淘汰 LRU（最近最少使用） |
| **volatile-lru** | 有过期时间的 key 中淘汰 LRU |
| **allkeys-lfu** | 所有 key 中淘汰 LFU（最不经常使用） |
| **volatile-lfu** | 有过期时间的 key 中淘汰 LFU |
| **allkeys-random** | 随机淘汰 |
| **volatile-random** | 有过期时间的 key 中随机淘汰 |
| **volatile-ttl** | 淘汰 TTL 最短的 key |

```redis
CONFIG SET maxmemory-policy allkeys-lru
```

### 3.1 Redis 的近似 LRU

Redis 不维护精确的全局 LRU 链表（开销太大）。而是**随机采样 N 个 key，淘汰其中最久未使用的**。`maxmemory-samples 5`（默认采样 5 个）。

### 3.2 LFU 的实现

Redis 4.0+ 的 LFU 通过 `lru` 字段的高 16 位存时间 + 低 8 位存计数器实现，计数器会随时间衰减。

## 4. 主从复制

```
┌─────────┐     全量同步(RDB)      ┌─────────┐
│  Master  │ ────────────────────→ │  Slave   │
│  (写)    │ ←──── 增量同步 ────── │  (读)    │
└─────────┘   (replication buf)   └─────────┘
```

### 4.1 同步流程

```
1. Slave: PSYNC ? -1  → Master
2. Master: +FULLRESYNC runid offset
3. Master: BGSAVE 生成 RDB → 发给 Slave
4. Slave: 加载 RDB
5. Master: 将 RDB 生成期间的写命令从 replication buffer 发给 Slave
6. Slave: 执行这些命令 → 追上 Master
```

## 5. 哨兵（Sentinel）

```
哨兵 = 监控 Master + 自动故障转移

┌──────────┐  ┌──────────┐  ┌──────────┐
│Sentinel 1│  │Sentinel 2│  │Sentinel 3│  ← 至少 3 个（奇数，用于投票）
└────┬─────┘  └────┬─────┘  └────┬─────┘
     └──────────────┼──────────────┘
                    │ 监控
               ┌────▼────┐
               │ Master  │ ← Master 挂了
               └────┬────┘
                    │ 选举一个 Slave 成为新 Master
               ┌────▼────┐
               │新 Master│
               └─────────┘
```

## 6. 集群（Cluster）

```
Redis Cluster: 16384 个哈希槽 (slot)

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ Node A       │  │ Node B       │  │ Node C       │
│ slots:0-5460 │  │slots:5461-   │  │slots:10923-  │
│              │  │     10922    │  │     16383    │
└──────────────┘  └──────────────┘  └──────────────┘

key → CRC16(key) % 16384 → 确定 slot → 确定 node
```

每个节点负责一部分 slot，数据分布在所有节点上。支持水平扩展（加节点 → 迁移 slot）。

## 7. 缓存一致性终极选择

```
实时性要求高？
├─ 是 → 读 DB + 写时删除缓存 (Cache Aside) + 延迟双删
└─ 否 → 定时任务同步 / Canal 监听 Binlog 异步更新缓存

可以容忍少量数据丢失？
├─ 是 → Write Behind（高吞吐）
└─ 否 → Cache Aside / Read Through
```

## 小结

| 概念 | 一句话 |
|------|--------|
| 缓存穿透 | 查不存在的 key → 布隆过滤 / 缓存空值 |
| 缓存击穿 | 热点 key 过期 → 互斥锁 / 逻辑过期 |
| 缓存雪崩 | 大量 key 同时过期 → 随机 TTL / 多级缓存 |
| Cache Aside | 先更新 DB → 再删除缓存 |
| 主从复制 | RDB 全量 + 增量同步 |
| 哨兵 | 监控 + 自动故障转移 |
| 集群 | 16384 槽，按 slot 分布数据 |
