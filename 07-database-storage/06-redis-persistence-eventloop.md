## 1. RDB——内存快照

### 1.1 工作原理

```
bgsave（后台保存）:
1. Redis 主进程 fork 出一个子进程
2. 子进程将当前内存数据写入临时 RDB 文件
3. 写入完成后，原子地 rename 替换旧 RDB 文件
4. 主进程完全不阻塞（利用 COW）
```

### 1.2 COW（写时复制）

fork 后父子进程共享同一物理内存。只有当主进程修改数据时，才复制被修改的那一页。

```
fork 前:  [物理页A] [物理页B] [物理页C]
fork 后:  父→A 父→B 父→C
          子→A 子→B 子→C   ← 全部共享，无拷贝

父进程修改 B:
          父→A 父→B'(新) 父→C
          子→A 子→B       子→C
          ↑ 只有 B 被复制
```

### 1.3 触发方式

```bash
# 手动触发
SAVE       # 主进程阻塞执行（不推荐）
BGSAVE     # 后台异步执行

# 自动触发（redis.conf）
save 900 1       # 900 秒内至少 1 次修改
save 300 10      # 300 秒内至少 10 次修改
save 60 10000    # 60 秒内至少 10000 次修改
```

### 1.4 优缺点

| 优点 | 缺点 |
|------|------|
| 文件紧凑，适合备份和灾难恢复 | 两次快照之间的数据会丢失 |
| 恢复速度快（直接加载） | fork 时内存翻倍风险（COW 期间） |

## 2. AOF——追加日志

### 2.1 工作原理

AOF 将每一条**写命令**追加到日志文件末尾（类似 WAL）。

```
SET key value  → AOF: *3\r\n$3\r\nSET\r\n$3\r\nkey\r\n$5\r\nvalue\r\n
INCR counter   → AOF: *2\r\n$4\r\nINCR\r\n$7\r\ncounter\r\n
```

恢复时：重放 AOF 中的所有命令。

### 2.2 fsync 策略

| 策略 | `appendfsync` | 性能 | 安全性 |
|------|:---:|:---:|:---:|
| always | 每条命令都 fsync | 最慢 | 最安全（最多丢 1 条） |
| **everysec** | 每秒 fsync 一次 | 中等 | 最多丢 2 秒数据（默认，推荐） |
| no | 由 OS 决定何时刷 | 最快 | 最不安全 |

### 2.3 AOF 重写

AOF 文件随着时间会变得巨大（如反复 INCR 同一个 key → 很多条 INCR 命令）。**AOF 重写**将其压缩为等价的最小命令集。

```
原 AOF:
SET counter 0
INCR counter  ... 100 次
INCR counter

AOF 重写后:
SET counter 102    ← 一条命令代替 102 条
```

重写也是通过 fork 子进程 + COW 完成，不阻塞主进程。

### 2.4 混合持久化（Redis 4.0+）

```
混合 AOF = RDB 快照 + AOF 增量

AOF 文件结构:
┌──────────────────┬───────────────────┐
│   RDB 格式快照    │  AOF 增量命令      │
│ (当前数据库状态)   │ (快照后的写命令)    │
└──────────────────┴───────────────────┘

恢复时：先加载 RDB 部分（快） → 再重放 AOF 增量
结合了 RDB 的快速恢复 + AOF 的数据安全性
```

## 3. Redis 事件循环与单线程模型

### 3.1 为什么单线程这么快？

```
Redis 单线程快的原因:
1. 所有数据在内存中 → 操作是纳秒级的
2. IO 多路复用 (epoll) → 一个线程处理成千上万的连接
3. 没有锁竞争 → 无上下文切换开销
4. 精心设计的数据结构 → 操作复杂度低
```

### 3.2 事件循环结构

```c
// Redis 事件循环（简化）
while (!stop) {
    // 1. 处理"时间事件"（定时任务，如过期键清理）
    processTimeEvents();

    // 2. epoll_wait 等待"文件事件"（客户端连接/请求）
    int num = aeApiPoll(tv);
    for (int i = 0; i < num; ++i) {
        // 3. 处理就绪的事件
        if (事件是"可读") processRead(fd);
        if (事件是"可写") processWrite(fd);
    }
}
```

### 3.3 Redis 6.0+ 多线程 IO

```
Redis 6.0 的多线程只用于网络 IO，命令执行仍是单线程:

┌──────────────┐
│ Main Thread   │ ← 命令执行（单线程！保证原子性）
├──────────────┤
│ IO Thread 1  │ ← 解析请求（read + 协议解析）
│ IO Thread 2  │ ← 解析请求
│ IO Thread 3  │ ← 写回响应（write）
└──────────────┘
```

默认关闭，需配置 `io-threads 4`。

## 4. 过期键删除策略

| 策略 | Redis 的做法 |
|------|-------------|
| 惰性删除 | 访问 key 时检查是否过期 → 过期则删除 |
| 定期删除 | 每 100ms 随机取一批 key 检查并删除超时的 |
| 定时删除 | ❌ Redis 不用（太耗 CPU） |

惰性删除 + 定期删除的组合保证性能和空间平衡。

## 小结

| 概念 | 一句话 |
|------|--------|
| RDB | fork 子进程 + COW，生成内存快照 |
| AOF | 追加写命令，everysec fsync |
| AOF 重写 | 压缩冗余命令，fork 子进程执行 |
| 混合持久化 | RDB 快照 + AOF 增量 |

| 多线程 IO | 只多线程处理网络，命令执行仍单线程 |
